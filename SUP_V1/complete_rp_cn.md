# Privacy-preserving Retrieval-augmented Generation Based on Federated Learning

## 题目

**基于联邦学习的隐私保护型检索增强生成：面向通信效率的超网络选择性参数上传机制**

## 1. Background

检索增强生成（Retrieval-Augmented Generation, RAG）已成为提升大语言模型（Large Language Models, LLMs）事实准确性、可追溯性与领域适应能力的重要技术路径。通过将外部知识库中的相关文档片段引入生成过程，RAG 能够有效缓解 LLM 的幻觉问题，并在金融、法律、医疗等垂直领域问答任务中展现出较强的应用潜力。

然而，在真实私有化场景中，RAG 的部署仍受到数据隐私和数据治理约束的限制。金融机构、法律机构或医疗机构通常拥有高价值但敏感的本地知识库，这些数据受到 GDPR 等法规或内部合规政策约束，难以直接集中到统一服务器中训练检索器或构建共享向量数据库。此外，向量表示本身也可能面临反向推断、成员推断或语义泄露风险。因此，如何在不暴露原始数据的前提下协同提升 RAG 检索性能，成为隐私保护型 RAG 的核心问题。

联邦学习（Federated Learning, FL）为该问题提供了一种可行范式。FL 允许多个客户端在数据不出域的前提下协同训练模型，服务器仅接收模型更新而非原始数据。将 FL 引入 RAG 后，各客户端可以基于本地私有语料训练检索器嵌入模型，并通过联邦聚合吸收其他客户端的知识，从而提升私有 RAG 系统在数据稀缺或 Non-IID 场景下的检索效果。

FedE4RAG 是该方向中具有代表性的工作。它将联邦学习、知识蒸馏（Knowledge Distillation, KD）和同态加密（Homomorphic Encryption, HE）结合起来，在不共享原始私有数据的情况下协同优化客户端侧 RAG 检索器。然而，现有 FedE4RAG 方案在通信效率方面仍存在明显不足：客户端通常需要在每轮训练中上传完整嵌入模型参数或完整梯度更新。随着检索器规模增大，该全量同步机制会产生较高通信成本；若进一步结合 HE，则加密后的参数传输和聚合开销也会显著增加。因此，本文关注的问题是：如何在维持隐私保护和检索性能的前提下，降低联邦 RAG 上游训练阶段的通信消耗。

## 2. Related Work

### 2.1 隐私保护与知识增强 RAG

为缓解 RAG 系统中的隐私风险，已有研究从差分隐私、知识图谱、可信执行环境等角度提出了多种方案。DPRAG 通过差分隐私机制在推理侧对输出 Token 加噪，以降低敏感记录泄露风险，但隐私预算会限制生成长度，并可能影响生成准确性。GraphRAG 通过知识图谱增强跨文档推理与全局知识组织能力，有助于缓解复杂问答中的知识冲突，但图谱构建成本较高，且通常依赖特定 schema 或领域知识工程。C-FedRAG 利用可信执行环境（Trusted Execution Environment, TEE）提供硬件级隔离保护，但其部署受到硬件依赖和内存限制影响，在大规模向量检索场景中存在扩展性挑战。

上述方法分别从推理侧保护、知识结构增强或硬件可信执行角度提升 RAG 的隐私性与可靠性，但对“多机构私有数据协同训练检索器”这一问题仍缺乏充分支持。尤其是在跨机构知识库无法集中共享的情况下，仅依靠本地训练往往难以获得足够鲁棒的检索器。

### 2.2 联邦学习与 RAG 的结合

联邦学习为跨机构私有 RAG 提供了去中心化训练方式。FLERAG 初步探索了在不集中数据的前提下利用 FL 提升 RAG 适用性，但其主要关注 FL 与 RAG 的框架融合，对复杂知识冲突和通信效率的讨论仍较有限。FeB4RAG 针对分布式向量数据库构建了联邦检索问答基准，有助于评估联邦搜索场景下的检索效果，但其重点在于基准设计与性能评估，而非通信或隐私协议优化。

FedE4RAG 进一步将 FL、KD 和 HE 引入垂直领域私有 RAG 场景，通过联邦协议在多客户端之间协同微调嵌入模型，并利用知识蒸馏缓解 Non-IID 数据下的本地模型偏移。该方案能够在原始数据不出域的前提下提升本地检索器能力，是当前隐私保护型联邦 RAG 的重要基础框架。然而，FedE4RAG 的通信机制仍接近传统联邦训练范式，即客户端频繁上传高维完整模型参数或梯度更新。当嵌入模型规模较大、客户端数量增加或启用 HE 加密时，该机制会导致显著的通信与计算开销。

### 2.3 通信高效联邦优化与超网络机制

为降低 FL 通信开销，已有研究提出了梯度压缩、稀疏上传、参数剪枝、低秩适配和客户端选择等方法。这些方法通常通过减少上传参数量、降低更新频率或压缩更新表示来提升通信效率。然而，在联邦 RAG 场景中，不同客户端的数据分布、检索语义和参数更新重要性可能存在差异，固定的压缩规则或随机稀疏策略可能难以稳定保留对检索器性能最关键的参数更新。

pFedLA 等个性化联邦学习方法表明，超网络可以根据客户端表示生成层级聚合权重，从而捕捉客户端异质性。本文借鉴其“客户端相关的动态参数评估”思想，但不直接采用其个性化模型生成目标。本文将超网络重新定义为参数重要性评估器和通信控制器，使其为每个客户端、每个模型参数块预测上传重要性，从而实现面向通信效率的选择性参数上传。

## 3. Current Challenges

### 3.1 FedE4RAG 框架流程

FedE4RAG 的核心流程可分为上游联邦嵌入学习和下游 RAG 评测两个阶段。在上游阶段，多个客户端分别持有本地私有训练数据，如查询、答案或参考文本。服务器初始化一个嵌入模型，并在每轮通信中选择客户端参与训练。客户端接收服务器下发的模型后，在本地使用查询-参考文本对进行训练。其本地训练目标通常包括检索对比学习损失和知识蒸馏损失：前者用于提升查询与正确参考文本之间的向量匹配能力，后者用于利用服务器教师模型的 logits 引导本地模型学习全局知识。

在当前仓库实现中，`FedE/main.py` 负责生成联邦任务并初始化 `flgo.algorithm.fedrag` 算法；`FedE/flgo/benchmark/fedrag_classification/core.py` 定义了 FedRAG 训练数据、任务划分和客户端损失计算；`fedrag.py` 中的 Server 与 Client 分别负责模型下发、本地训练、客户端回复和服务器端聚合。客户端训练完成后，服务器收集本地模型更新，并通过平均或加权平均更新全局模型。

在下游阶段，训练后的嵌入模型被用于 RAGTest 模块。`RAGTest/index.py` 根据测试语料构建向量索引，`RAGTest/retriever.py` 实现向量检索、BM25、融合检索等策略，`RAGTest/eval` 中的脚本计算 Hit@k、MRR、MAP、NDCG 等检索指标，以及 ROUGE、CHRF、METEOR 等生成指标。该流程用于验证上游联邦训练得到的检索器能否提升下游 RAG 检索与问答效果。

### 3.2 通信消耗过大的挑战

尽管 FedE4RAG 能够在隐私保护场景下提升检索器性能，但其默认通信模式仍存在明显瓶颈。客户端在每轮训练结束后通常需要上传完整嵌入模型参数或完整梯度更新。对于 BERT/BGE 风格的嵌入模型而言，参数量较大，完整上传会导致每轮训练产生较高带宽消耗。随着客户端数量、通信轮数和模型规模增加，通信开销会进一步放大。

该问题在引入同态加密后更为突出。HE 能够保护上传参数更新不以明文形式暴露给服务器，但加密后的参数体积和计算成本通常高于明文传输。如果仍对完整模型更新进行加密和上传，通信效率会成为限制 FedE4RAG 在真实带宽受限环境中部署的重要因素。

此外，完整参数上传并不一定是必要的。不同客户端、不同训练轮次、不同模型层对全局检索器性能的贡献并不均等。某些底层或低变化参数块的更新可能较小，直接上传会造成冗余通信；某些高层语义参数块或任务相关层的更新可能更重要，更值得优先传输。因此，FedE4RAG 当前面临的关键挑战可以概括为：**如何在不牺牲下游 RAG 检索效果的前提下，识别并上传更重要的参数更新，从而降低联邦训练通信开销。**

## 4. Research Objectives

本文研究目标是在 FedE4RAG 的隐私保护联邦 RAG 框架基础上，引入一种基于超网络的选择性参数上传机制，以降低上游联邦训练阶段的通信消耗，并尽量保持下游 RAG 检索性能。

具体目标包括三点。第一，设计一个参数重要性评估机制，使系统能够根据客户端特征和参数块更新统计动态判断哪些参数更值得上传。第二，在不改变 FedRAG 本地训练目标的前提下，将完整参数上传改为参数块级别的选择性上传，从通信阶段减少冗余更新。第三，将该机制与下游 RAG 评测流程连接起来，通过检索指标和生成指标验证通信压缩对实际 RAG 效果的影响。

本文理论上参考 pFedLA 中的超网络思想。pFedLA 使用超网络根据客户端信息生成模型层级聚合权重，以适应客户端异质性。本文并不直接沿用其个性化模型生成机制，而是将超网络用于参数重要性评估：输入客户端嵌入、参数块嵌入和历史更新统计，输出每个参数块的重要性分数，并据此选择 top-\(K\) 参数块上传。这样，超网络从“个性化模型生成器”转化为“通信控制器”，服务于隐私保护型联邦 RAG 中的通信效率优化。

## 5. The Method

### 5.1 总体方法与问题建模

PPRAG-FL 采用客户端-服务器结构。第 \(i\) 个客户端拥有私有数据集 \(D_i\)，其中包含查询与参考文本对 \((q,r)\)。系统目标是在不共享原始数据的条件下协同训练检索嵌入模型 \(f_\theta(\cdot)\)。客户端本地训练目标沿用 FedE4RAG 的设计：

\[
\mathcal{L}_i = \mathcal{L}_{\text{contrastive}} + \lambda \mathcal{L}_{\text{distill}}.
\]

其中，\(\mathcal{L}_{\text{contrastive}}\) 用于学习查询与参考文本之间的向量匹配关系，\(\mathcal{L}_{\text{distill}}\) 用于使客户端模型对齐服务器教师模型的 logits。与原有 FedE4RAG 不同的是，PPRAG-FL 在客户端本地训练完成后不上传完整更新：

\[
\Delta_i^t = \theta_i^t - \theta^t,
\]

而是仅上传由超网络筛选出的参数块更新：

\[
\Delta_{i,\mathcal{U}}^t = \{\Delta_{i,b}^t \mid b \in \mathcal{U}_i^t\}.
\]

这里 \(b\) 表示模型参数块，\(\mathcal{U}_i^t\) 表示客户端 \(i\) 在第 \(t\) 轮需要上传的参数块集合。

### 5.2 参数块划分与超网络重要性评估

为避免逐参数稀疏上传带来的索引开销和训练不稳定，本文采用参数块级别选择。对于 BERT/BGE 风格检索器，参数可划分为：

\[
\mathcal{B} = \{\text{embeddings}, \text{encoder.layer.0}, \ldots, \text{encoder.layer.11}, \text{pooler}\}.
\]

该划分方式符合 Transformer 嵌入模型结构，也便于分析不同层对检索任务的贡献。

本文引入轻量级超网络 \(H_\phi\) 预测参数块重要性。对于客户端 \(i\) 和参数块 \(b\)，超网络输入为：

\[
z_{i,b} = [e_i; e_b; s_{i,b}^{t-1}],
\]

其中 \(e_i\) 为客户端嵌入，\(e_b\) 为参数块嵌入，\(s_{i,b}^{t-1}\) 为上一轮该参数块的更新统计特征。超网络输出重要性分数：

\[
\alpha_{i,b}^t = H_\phi(z_{i,b}).
\]

客户端在本地训练后计算每个参数块的轻量统计信息，例如块更新的 \(L_2\) 范数、平均更新幅度和最大绝对更新值。服务器将归一化后的块更新范数作为训练超网络的监督信号：

\[
y_{i,b}^t = \frac{\|\Delta_{i,b}^t\|_2}{\max_{b' \in \mathcal{B}}\|\Delta_{i,b'}^t\|_2 + \epsilon}.
\]

超网络训练目标为：

\[
\mathcal{L}_{H} = \sum_{b \in \mathcal{B}} (H_\phi(z_{i,b}) - y_{i,b})^2.
\]

这种方式使超网络能够逐轮学习不同客户端、不同参数块的重要性模式，并在后续通信轮次中预测更合理的上传 mask。

### 5.3 选择性上传与稀疏聚合

在每轮训练开始前，服务器根据超网络输出的重要性分数为客户端构造 top-\(K\) 上传集合：

\[
\mathcal{U}_i^t = \text{TopK}_{b \in \mathcal{B}}(\alpha_{i,b}^t).
\]

训练初期可设置若干 warm-up 轮，客户端上传完整参数更新，以便服务器收集初始统计特征。warm-up 结束后，服务器向客户端下发当前模型与上传 mask；客户端仍执行完整本地训练，但只上传 mask 指定参数块的 delta 以及所有参数块的轻量统计信息。

服务器接收稀疏更新后，只对被上传的参数块执行聚合。对于参数块 \(b\)，设第 \(t\) 轮上传该块的客户端集合为 \(S_b^t\)，则：

\[
\theta_b^{t+1} =
\theta_b^t +
\sum_{i \in S_b^t}
\frac{|D_i|}{\sum_{j \in S_b^t}|D_j|}
\Delta_{i,b}^t.
\]

若某个参数块在当前轮没有被任何客户端上传，则服务器保持该块参数不变。该机制可以在不改变客户端本地训练目标的情况下，将通信内容从“完整模型更新”减少为“重要参数块更新”。

### 5.4 隐私保护与下游 RAG 推理

PPRAG-FL 保留 FedE4RAG 的隐私保护基本设定：原始数据始终保留在客户端本地。选择性上传进一步减少了传输参数范围，降低了潜在参数暴露面。若需要更强保护，可结合 FedE4RAG 中的 CKKS 同态加密思想，仅对被选中的参数块更新进行加密上传。由于加密对象从完整模型更新缩减为重要参数块，HE 带来的通信和计算开销也可同步下降。

上游联邦训练结束后，训练得到的检索器进入下游 RAG 流程。根据当前仓库实验逻辑，`RAGTest` 模块负责读取测试语料、构建向量索引、执行查询检索、调用 LLM 生成答案并计算评测指标。整体流程为：

\[
\text{Federated Retriever}
\rightarrow \text{Corpus Indexing}
\rightarrow \text{Query Embedding}
\rightarrow \text{Retrieval}
\rightarrow \text{LLM Generation}
\rightarrow \text{Evaluation}.
\]

通过该流程，可以验证上游选择性上传是否在减少通信消耗的同时保持下游 RAG 检索和生成质量。

## 6. Experiment Design

### 6.1 研究问题与对比方法

实验主要回答以下问题：PPRAG-FL 是否能显著降低上游联邦训练通信开销；在降低通信开销后，下游 RAG 检索性能是否保持稳定；上传参数块数量 \(K\) 如何影响通信效率与检索质量之间的权衡；超网络选择是否优于随机选择和固定规则选择。

实验拟设置以下对比方法。第一，FedE4RAG Full Upload，即原始完整参数上传基线。第二，Random Selective Upload，即在相同上传预算下随机选择参数块上传。第三，Static Top-Layer Upload，即固定上传最后 \(K\) 个 Transformer 层。第四，Delta-Norm Selection，即根据上一轮参数块更新范数选择 top-\(K\) 参数块。第五，PPRAG-FL，即本文提出的基于超网络的重要性评估与选择性上传方法。

### 6.2 数据集与实现设置

上游训练使用当前仓库 `FedE/train_data` 中的 FedE4RAG 风格训练数据，并将选定训练集整理为 `FedE/select_data.json`。每条样本包含查询、公司或类别标识以及参考文本，用于训练查询-参考文本匹配能力。联邦任务由 `FedE/main.py` 中的 `flgo.gen_task` 生成，训练 benchmark 为 `flgo.benchmark.fedrag_classification`。

下游评测使用 `RAGTest/data` 中的测试数据，例如 `data_50.json` 和 `data_100.json`。这些数据包含问题、答案、参考上下文及 golden reference ids，可用于评估检索结果与生成结果。若完整 Hugging Face 数据集可用，可进一步扩展到 FEDE4FIN 训练集和 RAG4FIN 测试集。

默认实验设置可与当前仓库保持一致：客户端数量为 5，通信轮数为 25，本地训练 epoch 为 1，batch size 为 8，学习率为 \(1 \times 10^{-5}\)。超网络初始设置可采用 embedding 维度 64、隐藏层维度 128、超网络学习率 \(5 \times 10^{-3}\)、warm-up 轮数 2、默认上传 top-\(K=3\) 个参数块。

### 6.3 评估指标与实验流程

实验从通信效率、检索性能和生成质量三个维度进行评估。通信效率主要报告 payload ratio：

\[
\text{Payload Ratio} =
\frac{\text{uploaded parameter count}}{\text{full trainable parameter count}}.
\]

此外记录每轮平均上传参数量、总上传参数量、平均 payload ratio，以及启用同态加密时的估计密文传输量。

检索性能采用 `RAGTest` 中已有指标，包括 Hit@k、Recall@k、MRR、MAP 和 NDCG。若启用答案生成流程，则进一步报告 ROUGE、CHRF、METEOR、WER、CER 和 Perplexity 等生成质量指标。

实验流程如下：首先准备 `FedE/select_data.json` 并训练完整上传基线；随后在相同数据划分、客户端数量和训练超参数下训练 PPRAG-FL；训练过程中记录每轮上传 mask、payload ratio、超网络损失和通信统计；上游训练结束后保存检索器模型；最后将模型路径传入 `RAGTest` 下游评测脚本，通过检索和生成指标比较各方法效果。

### 6.4 消融实验与预期结果

为验证方法各组成部分的作用，本文设计四类消融实验。第一，上传预算消融，设置 \(K=1,3,5,7\)，分析通信压缩率与检索性能之间的关系。第二，warm-up 轮数消融，比较 0、1、2、5 轮 warm-up 对超网络稳定性的影响。第三，选择策略消融，比较超网络选择、随机选择、固定高层选择和 delta-norm 启发式选择。第四，加密机制消融，比较启用与不启用 CKKS 同态加密时的通信和计算开销。

预期 PPRAG-FL 能够在 warm-up 后显著降低平均 payload ratio，并在相同通信预算下优于随机选择和固定层选择。若超网络能够准确识别对检索器更新贡献较大的参数块，则下游 Hit@k、MRR、MAP 和 NDCG 应接近或优于完整上传基线在低通信预算下的表现。与此同时，过小的上传预算可能导致关键更新遗漏，从而影响收敛速度和检索效果。因此，最终应综合通信压缩率与下游 RAG 指标确定合适的上传策略。
